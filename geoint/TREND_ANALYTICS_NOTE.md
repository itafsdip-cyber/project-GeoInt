# Trend Analytics Technical Note

## History storage approach
- Historical data is persisted in browser `localStorage` under the existing intelligence state key.
- A dedicated `historyStore` object now keeps:
  - `events`: normalized events as they pass through the current flow.
  - `incidents`: detected incidents generated by the existing incident detector.
- The history layer is intentionally lightweight and adapter-friendly:
  - append-only snapshot API (`appendHistorySnapshot`) for current usage.
  - read API (`loadHistoryStore`) to hydrate analytics at startup.
- Safety controls:
  - deduplication by stable identity (`id` / `incidentId`, with fallback fingerprints).
  - retention window (14 days).
  - hard caps (events: 4000, incidents: 1200).

## Trend calculations
Trend analytics compares the active window with the immediately preceding window of equal size:
- 1H vs previous 1H
- 6H vs previous 6H
- 24H vs previous 24H
- 7D vs previous 7D

Computed outputs:
- event volume delta
- incident volume delta
- category trends
- region trends
- actor frequency trends
- source activity trends

Trend labels are intentionally conservative:
- `UP`, `DOWN`, `FLAT` for directional movement.
- `SPIKE` for strong step-changes (roughly >=2x and meaningful absolute increase).
- `NEW` when current activity appears after zero activity in the prior comparison window.

## Watchlist-aware trend handling
- Trend outputs are cross-checked against watchlist items for dimensions: `region`, `category`, `actor`.
- If a watched term aligns with `UP`, `SPIKE`, or `NEW`, it is surfaced as a watchlist trend flag.

## Limitations
- Local browser storage is not durable across devices/browsers and can be cleared.
- History is bounded for performance; older events will age out.
- Trend logic is deterministic and heuristic-based, not inferential statistics.
- Indicator labels do not claim predictive certainty.

## Future backend evolution
For stronger longitudinal intelligence support:
1. Move history to backend persistence (PostgreSQL/Timescale/ClickHouse).
2. Store immutable event/incident snapshots with ingestion timestamps.
3. Add multi-tenant and analyst/user scoping.
4. Add query APIs for arbitrary windows and aggregations.
5. Add baseline seasonality models and confidence intervals where appropriate.
6. Add auditability/versioning for incident recalculation logic.
